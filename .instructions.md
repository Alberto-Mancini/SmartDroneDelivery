# SmartDroneDelivery - Development Guide

## 🏗️ Architettura Repository

```
main          → Produzione stabile (sempre funzionante)
  └── PRs da develop/feature/*

develop       → Integrazione (staging area)
  └── PRs da feature/*

feature/*     → Singola feature isolata
  (es: feature/improve-classifier, feature/new-algorithm)

bugfix/*      → Bug fix urgenti
  (es: bugfix/wind-cost-calculation)
```

## 📖 Workflow Sviluppo Locale

### Setup Iniziale
```bash
git clone https://github.com/Alberto-Mancini/SmartDroneDelivery.git
cd SmartDroneDelivery
git checkout develop  # Lavora su develop di default

python3 -m venv venv
source venv/bin/activate
pip install -r requirements.txt

python ml_classifier.py   # Allena classificatore
python main.py            # Test
```

### Per Iniziare una Feature
```bash
git checkout develop          # Partenza da develop
git pull origin develop       # Aggiorna
git checkout -b feature/my-feature

# Fai modifiche, committ regolari
git add src/file.py
git commit -m "feat: descrizione breve"

# Push quando pronto
git push origin feature/my-feature
```

### Commit Message Convention

| Prefisso | Uso | Esempio |
|----------|-----|---------|
| `feat:` | Nuova feature | `feat: Add RBFS algorithm` |
| `fix:` | Bug fix | `fix: Correct wind multiplier calculation` |
| `docs:` | Documentazione | `docs: Update algorithm descriptions` |
| `refactor:` | Riordino codice | `refactor: Simplify Grid initialization` |
| `test:` | Test e validazione | `test: Add KNN accuracy tests` |
| `perf:` | Performance | `perf: Optimize pathfinding for large grids` |

## 🔄 Pull Request Process

1. **Feature completa su local branch**
   ```bash
   git checkout feature/my-feature
   git status  # Verifica tutto è committed
   ```

2. **Push e crea PR**
   ```bash
   git push origin feature/my-feature
   ```
   Poi su GitHub → "Compare & pull request"

3. **PR Description Template**
   ```markdown
   ## Descrizione
   Cosa fa questa PR?

   ## Type
   - [ ] Feature
   - [ ] Bugfix
   - [ ] Refactor
   - [ ] Docs

   ## Testing
   Come testare? (comando, passi)

   ## Checklist
   - [ ] Codice testato localmente
   - [ ] README aggiornato
   - [ ] Nessun conflitto di merge
   ```

4. **Merge in develop**
   ```bash
   # Dopo approvazione su GitHub, merge via UI
   ```

5. **Quando merge da develop → main**
   - Ogni release stabile
   - Via PR + squash commit
   - Tag versione: `git tag v1.0.0 && git push --tags`

## 🧪 Testing Checklist

Prima di fare PR:

```bash
# 1. Testa il codice
python main.py
python ml_classifier.py

# 2. Verifica conformità
python -m py_compile *.py  # Syntax check

# 3. Esegui benchmark
python main.py  # Premi C per benchmark

# 4. Pulisci
rm -rf __pycache__ *.pyc
```

## 🔑 Regole Importanti

❌ **VIETATO**
- Pushare direttamente su `main`
- Mergare feature non testate
- Committ con messaggi generici ("fix", "update")
- Includere file grandi non gestiti da Git LFS

✅ **OBBLIGATORIO**
- Ogni feature su proprio branch
- Commits leggibili e descrittivi
- Testare prima di PR
- Aggiornare README se necessario

## 💡 Tips Professionali

### Aggiorna develop regolarmente
```bash
git checkout develop
git pull origin develop
```

### Sincronizza feature con develop aggiornato
```bash
git checkout feature/my-feature
git rebase origin/develop
# o merge se preferisci
git merge origin/develop
```

### Correggi ultimo commit non pushato
```bash
git commit --amend --no-edit
```

### Visualizza lo stato della repo
```bash
git log --oneline --graph --all
```

### Cleanup branch locale (dopo merge)
```bash
git branch -d feature/my-feature
```

## 📞 Domande Comuni

**Q: Quale branch uso per iniziare?**
A: Sempre `develop` come base. Main è solo per release stabile.

**Q: Posso fare feature da feature?**
A: No. Sempre da `develop`.

**Q: E se breck qualcosa?**
A: Git salva tutto. Usa `git reflog` + `git reset --hard <commit>` per rollback.

**Q: Come merge senza creare commit fastidioso?**
A: Su GitHub usa "Squash and merge" quando fai PR.

**Q: Versioning?**
A: Semantic versioning: `v1.0.0` = major.minor.patch
